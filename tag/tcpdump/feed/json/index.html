I recently had a situation where I needed to constantly listen to network traffic, while being mindful of not filling up the disk. Fortunately tcpdump has a nice built in way of doing that:

\n
tcpdump -w tcp.pcap -s0 -C 1500 -W 15
\n

That breaks down as:

\n

– -w tcp.pcap: write to files that all start with the name tcp.pcap
\n– -s0: save the whole packet
\n– -C 1500: limit each capture file to 1,500 MBs
\n– -W 15: limit the number of capture files to 15

\n

On disk you’ll get the following 15 files:

\n
tcp.pcap00\r\ntcp.pcap01\r\ntcp.pcap02\r\ntcp.pcap03\r\ntcp.pcap04\r\ntcp.pcap05\r\ntcp.pcap06\r\ntcp.pcap07\r\ntcp.pcap08\r\ntcp.pcap09\r\ntcp.pcap10\r\ntcp.pcap11\r\ntcp.pcap12\r\ntcp.pcap13\r\ntcp.pcap14\r\n
\n

Once it finishes with tcp.pcap14 is cycles back through the whole thing, over writing one file at a time, starting with tcp.pcap00.

\n

Exactly what period of time this will cover depends on how much network traffic you are capturing. It could be 5 minutes or 15, and vary between each file.

Capturing packets on iOS devices using remote virtual interface:

\n

\niOS 5 added a remote virtual interface (RVI) facility that lets you use OS X packet trace programs to capture traces from an iOS device. The basic strategy is:

\n

– Connect your iOS device to your Mac via USB.
\n– Set up an RVI for that device. This creates a virtual network interface on your Mac that represents the iOS device’s networking stack.
\n– Run your OS X packet trace program, and point it at the RVI created in the previous step.\n

\n

I tried this out, hardest part was looking up the UDID for my phone again.
